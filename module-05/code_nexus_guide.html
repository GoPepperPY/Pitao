01 · The big idea

Why this is hard, and what saves you

Imagine you have to write code that processes numbers, text, and log entries. The naïve approach is a wall of if isinstance(x, int): … elif isinstance(x, str): … scattered through your program. Every time a new data type appears, you hunt down every one of those if-ladders and edit them. It is fragile and it does not scale.

Object-oriented design offers a different deal. You define one common interface — a promise that says "anything that processes data can be asked to validate, ingest, and output." Then each data type gets its own class that keeps that promise in its own way. The surrounding code only ever talks to the promise, never to the concrete class. Add a new data type later and the old code keeps working untouched.

The thesis of the whole projectCode that depends on an interface instead of a concrete type can be extended without being rewritten. The three exercises are just three increasingly ambitious demonstrations of that single principle.

Four words name the techniques that make this work, and you will be asked about all four at the defense:

Abstract class

A class that defines the promise (the method names + signatures) but refuses to be used directly. It is a contract, not a worker.

Method overriding

Each subclass supplies its own body for a promised method — same name, specialised behaviour.

Inheritance

Subclasses get the shared, already-written parts for free (here: output) and only write what differs.

Polymorphism

One variable, many possible concrete types. The right overridden method runs automatically at runtime.

03 · The OOP toolkit

Abstract classes & the ABC module

An abstract class is a base class that is intentionally incomplete. It declares which methods every subclass must have, but doesn't (necessarily) say how they work. You cannot create an instance of it directly — trying to do so is an error. That's the point: it exists only to be inherited from.

In Python this comes from the standard abc module (one of your two authorised imports). You inherit from ABC and mark the must-implement methods with the @abstractmethod decorator.

# illustrative ONLY — a different world, not the project
from abc import ABC, abstractmethod

class Sensor(ABC):
    @abstractmethod
    def read(self) -> float:
        """Every sensor must know how to read. No body here."""

    def label(self) -> str:
        # a NORMAL (concrete) method: shared by all sensors
        return "sensor reading"

s = Sensor()   # TypeError: can't instantiate abstract class

Two things to notice, because the subject relies on both:

  • An abstract class can mix abstract methods (a body-less promise each subclass must fulfil) with concrete methods (a finished method every subclass inherits as-is).
  • A subclass becomes usable only once it has overridden every abstract method. Miss one, and that subclass is itself still abstract and can't be instantiated.
Map this onto Exercise 0DataProcessor is the abstract class. validate and ingest are the two abstract methods (promises). output is the one concrete method (written once, inherited by all three processors). The three processors are the subclasses that fulfil the promises.
04 · The OOP toolkit

Method overriding

Overriding means a subclass defines a method with the same name as one in its parent, replacing the parent's version for instances of that subclass. The parent says what; each child decides how.

# illustrative ONLY
class Thermometer(Sensor):
    def read(self) -> float:        # overrides Sensor.read
        return 21.4

class Barometer(Sensor):
    def read(self) -> float:        # overrides it differently
        return 1013.2

The same method name, two completely different bodies. When you later call x.read(), Python looks at the actual object behind x and runs its version. You'll override validate and ingest three times each — once per processor — and each override encodes what "valid data" and "ingest this" mean for that specific type.

Inheritance = code reuseBecause output is written once in the base class and never overridden, all three processors share it for free. That's the "code reuse" the defense note asks you to articulate: write the common part once, specialise only what differs.
05 · The OOP toolkit

Subtype polymorphism

"Poly-morphism" = "many forms." A single variable typed as the base class can, at runtime, hold any subclass instance — and calling a method on it runs the correct overridden version automatically. You write the loop once, against the base type, and it works for every present and future subclass.

# illustrative ONLY
sensors: list[Sensor] = [Thermometer(), Barometer()]
for sensor in sensors:
    print(sensor.read())   # 21.4 then 1013.2 — no isinstance, no if-ladder

Notice what's missing: no if isinstance(...), no type checks. The loop doesn't know or care which concrete sensor it's holding. That is the heart of Exercise 1's DataStream: it keeps a list of DataProcessor objects and asks each one, polymorphically, whether it can handle a piece of data — never branching on concrete type.

Question — answer before continuingThe subject's own reflection question: how does polymorphism let DataStream handle different data types without knowing their implementations? Try it, then expand the hint.
Show the shape of a good answer

Because DataStream only depends on the base interface (validate/ingest), it can call those methods on any object that is-a DataProcessor. Python dispatches to each object's own overridden method at runtime. Benefits: the stream code never changes when you add a processor (open for extension, closed for modification); no brittle type-checking ladder; each type's rules live in one place. Say it in your words — don't memorise this.

06 · The OOP toolkit

Type annotations & the deliberate mypy warning

Your other authorised import is typing. Every method needs full type annotations, checked with mypy. Most of this is routine — until you hit the subtle bit the subject builds an entire requirement around. Read this section twice; it is the single most asked-about idea at the defense.

The two signatures that look contradictory

MethodBase class signatureIn the subclasses
validatevalidate(self, data: Any) -> boolSame signature, unchanged
ingestingest(self, data: Any) -> NoneNarrowed to the types it accepts

Why the difference? validate's job is to be handed anything at all and answer "can I deal with this?" So it must keep accepting Any in every subclass — it can't refuse a type before inspecting it. ingest is different: the subject explicitly wants each processor's ingest to advertise the concrete types it expects (e.g. the numeric one's parameter is annotated for ints/floats and lists of them), so the signature documents the contract.

Why that triggers a warning — and why it's correct here

There is a design rule called the Liskov Substitution Principle: a subclass should be usable anywhere the base class is. For method parameters, that means an override may safely accept more types than the parent (widen), but not fewer (narrow). Narrowing breaks substitutability — someone holding the base type believes they can pass Any, but the subclass secretly rejects most of it.

# illustrative ONLY — the same shape as the trap
class Animal:
    def feed(self, food: Any) -> None: ...

class Cat(Animal):
    def feed(self, food: Fish) -> None: ...   # narrower param
# mypy: "Argument 1 ... incompatible with supertype" — by design here
The subject ASKS for this warningExercise 0 says testing ingest with invalid data and no prior validate "will leave you with a mypy warning, on purpose." That warning is the visible symptom of narrowing ingest's parameter type. You are meant to see it — it proves you understood the contract well enough to deliberately step outside it. At the defense, be ready to explain exactly this: narrowing a parameter type violates Liskov substitution, which is why a static checker flags it.

The intended runtime safety net is separate from the type checker: if someone skips validate and feeds garbage straight into ingest, your code should raise an exception rather than silently corrupt the stream. (The example shows Got exception: Improper numeric data.) Static typing documents intent; the exception enforces it at runtime.

The output return type

output(self) -> tuple[int, str] hands back two things together: an int (a processing rank / position) and a str (the data, already turned into text). Same shape for every processor, so downstream code can treat all outputs identically — another payoff of the shared interface.

07 · The OOP toolkit

Duck typing & Protocol

Exercise 2 introduces a second, looser way to define an interface. The phrase is "if it walks like a duck and quacks like a duck, it's a duck" — duck typing cares about what an object can do, not what it inherits from.

ABC (nominal)Protocol (structural)
Compatibility decided byInheritance: you must subclass itShape: you just need the right methods
Used inEx 0 — DataProcessorEx 2 — ExportPlugin
Mental model"You signed the contract""You happen to fit the mould"

A Protocol (from typing) defines a method shape that any class can satisfy without explicitly inheriting from it. As long as a class has a process_output(self, data: list[tuple[int, str]]) -> None method, it counts as an ExportPlugin — whether it's the CSV plugin, the JSON plugin, or one you write tomorrow.

# illustrative ONLY
from typing import Protocol

class Renderer(Protocol):
    def render(self, text: str) -> None: ...

class ConsoleRenderer:        # note: does NOT inherit Renderer
    def render(self, text: str) -> None:
        print(text)               # yet it counts as a Renderer
Why two interface styles in one project?It's a teaching contrast. ABC gives you a strict family tree where reuse and overriding live. Protocol gives you a plug-in port where unrelated classes can dock just by matching a method shape. Knowing when to reach for each is exactly what the defense is probing.

09 · Exercise 0 — ex0/data_processor.py

Build the foundation: one base, three workers

This is where the entire architecture is born. You are building the abstract base plus three concrete processors. Nothing routes data yet — that's Ex 1. Here you just prove each processor can validate, ingest, and output on its own.

The architecture you must produce

DataProcessor (ABC) └─ is inherited by ─► NumericProcessor TextProcessor LogProcessor

The three methods on the base

MethodKindResponsibility
validate(self, data: Any) -> boolabstract"Can I handle this?" Accepts anything, returns yes/no. Same signature in all subclasses.
ingest(self, data: Any) -> NoneabstractAccept and store the data. Subclass signatures narrow to their real types. Raise if handed invalid data.
output(self) -> tuple[int, str]concreteHand back the oldest stored item + its rank, and remove it. Written once, inherited by all.

What each processor accepts

Numeric

Ingests int, float, and lists mixing both. Converts each value to a string before storing.

Text

Ingests str and lists of strings. Stores them as-is (already text).

Log

Ingests dict of string→string, and lists of such dicts. Converts each to a string like LEVEL: message.

"Oldest item + rank" — what storage shape does that imply?

The output method must return the oldest piece of data and then remove it. "Oldest out first" is the definition of a FIFO queue. A plain Python list works fine: append on ingest, take from the front on output. The "processing rank" is a counter of how many items this processor has ever accepted — it's the position/order, which is why outputs in the examples are numbered 0, 1, 2…

oldest · rank 0 · ► leaves first on output()
rank 1
rank 2
newest · rank 3 · ◄ arrived last via ingest()
A FIFO queue. output() pulls from the top (oldest); ingest() pushes to the bottom (newest).
Decide for yourself, then justify itShould the rank reset, or keep climbing across the processor's whole life? Look closely at how the example numbers items in Ex 1/Ex 2 (it keeps a total processed count separate from remaining). Reverse-engineer the intended behaviour from the expected output — that skill of reading sample output as a spec is itself being tested.

The required tests (bottom of the example output)

  1. Instantiate each of the three processors.
  2. Call validate with both good and bad data for each, and print the boolean.
  3. Call ingest with bad data without validating first and confirm it raises (this is the line that produces the intentional mypy warning).
  4. Ingest several valid items, then pull them back with output, printing each.
QuestionThe expected output shows validate('42'): True for the numeric processor but validate('42'): False for the text processor. What does that tell you about whether validate looks at the value's type or its content? (Hint: 42 there is shown as the integer the program holds, and each processor judges by type.)
10 · Exercise 1 — ex1/data_stream.py

Route a mixed stream — polymorphism in action

Now you reuse Ex 0 and add a coordinator: DataStream. It holds a list of registered processors and, for each element in an incoming mixed list, finds the one processor that can handle it.

Three methods to implement on DataStream

MethodJob
register_processor(self, proc: DataProcessor) -> NoneAdd a processor to the internal list. Note the parameter is typed as the base class — that's polymorphism enabling you to register any subclass.
process_stream(self, stream: list[Any]) -> NoneFor each element, ask the registered processors (via validate) which one accepts it, then ingest it there. If none accepts it, print an error and move on — don't crash.
print_processors_stats(self) -> NonePrint, per processor, total items ever processed and how many remain stored.

The routing logic, in plain English

take element► ask each processor: validate?► first yes → ingest there / all no → print error, continue

Look carefully at the example: with only the numeric processor registered, the strings, the list of dicts, and the list of strings all produce DataStream error - Can't process element…, while the numbers go through. After registering the other two and resending the same batch, everything is accepted. This is the whole lesson made visible: the same stream code behaves differently purely based on which processors exist — it never changed.

Read the counters carefullyThe flat list [3.14, -1, 2.71] counts as 3 numeric items (the processor flattens lists into individual stored items), which is why "total 4 items processed" appears after a batch containing one list-of-3 plus one lone 42. Your ingest must expand list inputs into individual stored entries. Trace every number in the example by hand until the counts match — that's how you confirm your model is right.
Question (the subject asks this directly)Why can process_stream work without knowing any concrete processor type, and what are the benefits? Answer in your own words; see §05 for the shape of a strong response. Be ready to also say what would change if you added, say, a BinaryProcessor — answer: only a new class + one register_processor call; the stream code is untouched.
11 · Exercise 2 — ex2/data_pipeline.py

Add the output side — a plug-in port via duck typing

The input half (processors + routing) is done. Now you build the output half: a pluggable export system. Instead of an ABC family tree, this uses a Protocol (§07) so unrelated export classes can dock just by matching a method shape.

What to implement

PieceDetail
ExportPluginA class inheriting Protocol. Defines one required method: process_output(self, data: list[tuple[int, str]]) -> None. Note the parameter type exactly matches a list of what DataProcessor.output returns.
output_pipeline(self, nb: int, plugin: ExportPlugin) -> NoneNew method on DataStream. After processing, it consumes nb items from every registered processor (using their output) and hands them to the plugin's process_output.
CSV plugin & JSON pluginTwo classes that each fit the mould (have process_output). They build valid CSV / JSON strings by hand — no csv/json import allowed.

What the example output is telling you

For CSV, the values from a batch are joined on one comma-separated line per processor (3.14,-1,2.71). For JSON, each consumed item becomes a key/value pair keyed by its rank ({"item_3": "42", "item_4": "21", …}). The int half of each tuple is the rank that becomes the JSON key; the str half is the value. Study how the JSON keys continue climbing across batches — that confirms rank is a lifetime counter, not a per-batch reset, and ties back to the decision you made in §09.

The conceptual payoffThe same output_pipeline works with the CSV plugin or the JSON plugin without a single change, because it depends only on the process_output shape — not on which class provides it. Input side = ABC polymorphism; output side = Protocol duck typing. You've now demonstrated both ways to "depend on an interface, not a type."
Hand-rolling the formatsThink about edge cases the libraries would normally handle: what if a value contains a comma (CSV) or a quote/backslash (JSON)? You may not need to solve every case for this subject, but being able to discuss them at the defense shows you understand why those libraries exist.
